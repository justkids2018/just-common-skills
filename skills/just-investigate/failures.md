# 失败模式记录 (Failure Patterns)

本文档记录 `just-investigate` 技能在实际使用中遇到的失败案例和解决方案。

---

## 失败案例 1: 没有复现问题就开始调查

### 场景
用户报告了一个 bug，AI 没有先复现问题就开始查看代码。

### 表现
- 无法确认问题是否真的存在
- 修复了"可能的问题"但实际问题还在
- 浪费时间在错误的方向

### 根因
- 急于找到答案
- 没有验证问题描述
- 过度依赖用户的描述

### 解决方案
- **复现优先原则**:
  1. 理解问题描述（什么场景？什么表现？）
  2. 尝试复现问题（本地环境、测试环境）
  3. 确认问题确实存在
  4. 记录复现步骤
  5. 开始调查
- **如果无法复现**:
  - 询问更多细节（环境、数据、操作步骤）
  - 检查日志
  - 考虑环境差异

### 预防措施
- 在 SKILL.md 中明确：必须先复现问题
- 要求 AI 输出复现步骤

## 失败案例 2: 只看代码，不看日志

### 场景
AI 只是阅读代码试图找出问题，没有查看日志、错误信息、堆栈跟踪。

### 表现
- 猜测问题原因
- 修复了"可能的问题"但实际问题还在
- 忽略了明显的错误信息

### 根因
- 认为"看代码就能找到问题"
- 没有意识到日志的重要性
- 不知道如何查看日志

### 解决方案
- **日志优先策略**:
  1. 查看应用日志（stdout, stderr）
  2. 查看错误日志（error.log）
  3. 查看系统日志（syslog, journalctl）
  4. 查看 CI/CD 日志
  5. 查看浏览器控制台（前端问题）
- **关键信息**:
  - 错误信息（error message）
  - 堆栈跟踪（stack trace）
  - 时间戳（何时发生）
  - 上下文（请求参数、用户操作）

### 预防措施
- 在 SKILL.md 中添加日志检查步骤
- 提供日志查看命令清单

## 失败案例 3: 修改代码试图"修复"，但没有理解根因

### 场景
AI 看到一个错误后立即修改代码，但没有理解为什么会出错。

### 表现
- 修复了表面问题，根因还在
- 引入了新的 bug
- 同样的问题在其他地方再次出现

### 根因
- 急于修复
- 没有深入分析
- 只看症状，不看病因

### 解决方案
- **根因分析流程**:
  1. **观察症状** - 问题的表现是什么？
  2. **收集证据** - 日志、错误信息、堆栈跟踪
  3. **提出假设** - 可能的原因是什么？
  4. **验证假设** - 通过实验或代码分析验证
  5. **定位根因** - 真正的原因是什么？
  6. **设计修复** - 如何修复根因？
  7. **验证修复** - 问题是否真的解决？
- **5 个为什么法**:
  - 为什么会出现这个错误？
  - 为什么会有这个条件？
  - 为什么没有处理这个情况？
  - 为什么测试没有发现？
  - 为什么设计时没有考虑？

### 预防措施
- 在 SKILL.md 中强调：先分析根因，再修复
- 要求 AI 输出根因分析

## 失败案例 4: 孤立地看问题，没有考虑系统上下文

### 场景
AI 只关注出错的代码片段，没有考虑整个系统的状态和交互。

### 表现
- 忽略了依赖服务的问题
- 忽略了配置问题
- 忽略了环境差异

### 根因
- 只看局部，不看全局
- 没有理解系统架构
- 没有检查依赖

### 解决方案
- **系统化调查清单**:
  - [ ] **代码层** - 逻辑错误、边界条件
  - [ ] **配置层** - 环境变量、配置文件
  - [ ] **依赖层** - 第三方服务、数据库、缓存
  - [ ] **基础设施层** - 网络、磁盘、内存
  - [ ] **数据层** - 数据格式、数据完整性
- **检查依赖**:
  - 数据库是否正常？
  - 缓存是否正常？
  - 第三方 API 是否正常？
  - 网络是否通畅？

### 预防措施
- 在 SKILL.md 中添加系统化调查清单
- 要求 AI 检查所有层次

## 失败案例 5: 没有使用调试工具

### 场景
AI 只是阅读代码和日志，没有使用调试器、性能分析工具等。

### 表现
- 无法定位具体的执行路径
- 无法查看变量值
- 无法分析性能瓶颈

### 根因
- 不知道有哪些调试工具
- 认为"看代码就够了"
- 不会使用调试工具

### 解决方案
- **调试工具清单**:
  - **调试器** - gdb, lldb, pdb, node inspect
  - **性能分析** - perf, py-spy, Chrome DevTools
  - **网络分析** - tcpdump, Wireshark, curl -v
  - **系统分析** - strace, lsof, netstat
  - **日志分析** - grep, awk, jq
- **何时使用**:
  - 需要查看变量值 → 调试器
  - 需要分析性能 → 性能分析工具
  - 需要分析网络 → 网络分析工具
  - 需要分析系统调用 → strace

### 预防措施
- 在 SKILL.md 中添加调试工具清单
- 根据问题类型推荐工具

## 失败案例 6: 没有记录调查过程

### 场景
AI 进行了大量调查，但没有记录过程和发现。

### 表现
- 忘记了之前的发现
- 重复调查相同的路径
- 无法向他人解释调查过程

### 根因
- 认为"记住就行"
- 没有意识到记录的重要性
- 调查过程混乱

### 解决方案
- **调查日志模板**:
  ```markdown
  ## 问题描述
  [简要描述问题]
  
  ## 复现步骤
  1. ...
  2. ...
  
  ## 调查过程
  ### 假设 1: [描述假设]
  - 检查了: [检查内容]
  - 发现: [发现内容]
  - 结论: [验证结果]
  
  ### 假设 2: [描述假设]
  - ...
  
  ## 根因
  [最终定位的根因]
  
  ## 修复方案
  [如何修复]
  
  ## 验证
  [如何验证修复有效]
  ```
- **记录到文件** - `investigation-<issue-id>.md`

### 预防措施
- 在 SKILL.md 中要求记录调查过程
- 提供调查日志模板

## 反模式总结

### ❌ 不要做
1. **不复现就调查** - 无法确认问题
2. **只看代码不看日志** - 忽略明显的错误信息
3. **急于修复** - 没有理解根因
4. **孤立地看问题** - 忽略系统上下文
5. **不使用调试工具** - 只靠"看"
6. **不记录过程** - 忘记发现，重复调查

### ✅ 应该做
1. **先复现问题** - 确认问题存在
2. **日志优先** - 查看所有相关日志
3. **根因分析** - 使用 5 个为什么法
4. **系统化调查** - 检查所有层次
5. **使用工具** - 调试器、性能分析、网络分析
6. **记录过程** - 使用调查日志模板
