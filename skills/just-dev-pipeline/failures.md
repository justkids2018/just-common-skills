# 失败模式记录 (Failure Patterns)

本文档记录 `just-dev-pipeline` 技能在实际使用中遇到的失败案例和解决方案。

## 目的

- 记录真实失败场景，而非理论风险
- 提供可操作的解决方案
- 帮助用户避免相同的坑

---

## 失败案例 1: 跳过需求澄清导致返工

### 场景
用户直接说"实现 XX 功能"，AI 没有充分澄清需求就开始编码。

### 表现
- 实现到一半发现需求理解错误
- 需要大量返工
- 浪费时间和 token

### 根因
- 用户描述模糊（"添加登录功能"）
- AI 没有追问关键细节（OAuth? 本地账号? 第三方登录?）
- 直接跳到实现阶段

### 解决方案
- **强制需求澄清阶段** - 即使用户说"直接实现"，也要先问清楚
- **关键问题清单**:
  - 这个功能的核心目标是什么？
  - 有哪些边界条件？
  - 预期的用户体验是什么？
  - 有没有现有代码需要兼容？

### 预防措施
- 在 SKILL.md 中明确：第一步必须是需求澄清，不可跳过
- 如果用户坚持跳过，至少要列出假设并让用户确认

---

## 失败案例 2: 分析阶段读取文件过多导致 context 溢出

### 场景
在代码/文档分析阶段，AI 读取了大量不相关的文件。

### 表现
- Context window 快速填满
- 后续实现阶段没有足够空间
- 需要重新开始会话

### 根因
- 没有先用 `find` 或 `grep` 定位相关文件
- 直接读取整个目录
- 没有评估文件大小

### 解决方案
- **分层搜索策略**:
  1. 先用 `find` 找到候选文件
  2. 用 `grep` 确认相关性
  3. 只读取真正需要的文件
- **优先读取小文件** - README, 配置文件, 接口定义
- **大文件分段读取** - 使用 `offset` 和 `limit` 参数

### 预防措施
- 在 SKILL.md 中添加"分析预算"概念
- 建议：分析阶段不超过 20 个文件读取

---

## 失败案例 3: 实现阶段没有增量验证

### 场景
AI 一次性实现了多个文件的修改，最后才运行测试。

### 表现
- 测试失败，但不知道哪个改动导致的
- 需要逐个回滚排查
- 浪费大量时间

### 根因
- 没有"实现一点，验证一点"的习惯
- 过度自信，认为一次性改对
- 没有增量提交

### 解决方案
- **增量实现策略**:
  1. 实现一个小功能
  2. 立即运行相关测试
  3. 通过后再继续下一个
- **每个增量都提交** - 方便回滚
- **失败时立即停止** - 不要继续堆代码

### 预防措施
- 在任务拆解时，确保每个子任务都可独立验证
- 在 SKILL.md 中强调"增量 + 验证"循环

---

## 失败案例 4: 自审阶段走过场

### 场景
AI 在自审阶段只是简单过了一遍代码，没有真正发现问题。

### 表现
- 提交后被人工审查发现明显问题
- 需要重新修复
- 降低信任度

### 根因
- 自审清单不够具体
- 没有真正运行测试
- 没有检查边界条件

### 解决方案
- **强制自审清单**:
  - [ ] 所有测试通过（包括新增测试）
  - [ ] 边界条件已覆盖
  - [ ] 错误处理已添加
  - [ ] 文档已更新
  - [ ] 没有 TODO 或 FIXME
- **必须运行测试** - 不能只是"看起来没问题"
- **检查 git diff** - 确保没有意外改动

### 预防措施
- 在 SKILL.md 中添加详细的自审清单
- 要求 AI 输出自审结果（不是"看起来没问题"，而是"运行了 X 个测试，全部通过"）

## 失败案例 5: 文档收口被遗忘

### 场景
功能实现完成，测试通过，提交了 PR，但忘记更新文档。

### 表现
- 其他开发者不知道新功能
- 文档和代码不一致
- 需要补充文档的 PR

### 根因
- 文档更新不在核心工作流中
- 认为"代码即文档"
- 时间紧迫时跳过文档

### 解决方案
- **文档作为交付物的一部分** - 不是可选项
- **在提交前检查**:
  - README 是否需要更新？
  - API 文档是否需要更新？
  - CHANGELOG 是否需要添加条目？
- **使用 `just-document-release`** - 专门的文档同步技能

### 预防措施
- 在 SKILL.md 的第 6 步明确：文档收口是必须的
- 提供文档更新清单

## 反模式总结

### ❌ 不要做
1. **跳过需求澄清** - "我知道你想要什么"
2. **一次性读取大量文件** - Context 是有限的
3. **大爆炸式实现** - 一次改 10 个文件
4. **走过场的自审** - "看起来没问题"
5. **忘记文档** - "代码就是文档"

### ✅ 应该做
1. **充分澄清需求** - 问清楚再动手
2. **精准定位文件** - 先搜索，再读取
3. **增量实现 + 验证** - 小步快跑
4. **认真自审** - 运行测试，检查清单
5. **文档同步** - 代码和文档一起更新

## 如何使用本文档

1. **实施前** - 阅读失败案例，避免重蹈覆辙
2. **遇到问题时** - 查找类似场景，参考解决方案
3. **实施后** - 如果遇到新的失败案例，添加到本文档
