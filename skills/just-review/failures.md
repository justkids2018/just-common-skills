# 失败模式记录 (Failure Patterns)

本文档记录 `just-review` 技能在实际使用中遇到的失败案例和解决方案。

---

## 失败案例 1: 只看新增代码，忽略上下文

### 场景
AI 只审查了 diff 中的新增行，没有理解周围的代码逻辑。

### 表现
- 遗漏了与现有代码的冲突
- 没有发现重复代码
- 没有识别出破坏性改动

### 根因
- 只看 `git diff`，没有读取完整文件
- 没有理解改动的上下文
- 没有检查调用方

### 解决方案
- **上下文审查策略**:
  1. 先看 diff，了解改动范围
  2. 读取被修改文件的完整内容
  3. 查找调用方（使用 `grep` 或 LSP）
  4. 理解改动对整个模块的影响
- **关键问题**:
  - 这个改动会影响哪些调用方？
  - 是否有重复代码可以复用？
  - 是否破坏了现有的 API 契约？

### 预防措施
- 在 SKILL.md 中要求：必须读取完整文件，不能只看 diff
- 提供上下文审查清单

## 失败案例 2: 走过场的审查 - "看起来没问题"

### 场景
AI 快速浏览代码后说"看起来没问题"，但实际有明显的逻辑错误。

### 表现
- 遗漏了空指针检查
- 没有发现边界条件问题
- 没有识别出性能问题

### 根因
- 审查不够深入
- 没有使用审查清单
- 过度自信

### 解决方案
- **强制审查清单**:
  - [ ] **逻辑正确性** - 代码逻辑是否正确？
  - [ ] **边界条件** - 空值、空数组、超长输入是否处理？
  - [ ] **错误处理** - 异常是否捕获和处理？
  - [ ] **性能** - 是否有 N+1 查询、无限循环？
  - [ ] **安全** - 是否有 SQL 注入、XSS 风险？
  - [ ] **可维护性** - 代码是否清晰易懂？
  - [ ] **测试覆盖** - 是否有对应的测试？
- **每个清单项都要有具体发现** - 不能只是打勾

### 预防措施
- 在 SKILL.md 中添加详细的审查清单
- 要求 AI 输出每个清单项的审查结果

---

## 失败案例 3: 没有运行代码就审查

### 场景
AI 只是静态阅读代码，没有实际运行或测试。

### 表现
- 遗漏了运行时错误
- 没有发现配置问题
- 没有验证改动是否真的工作

### 根因
- 认为"代码审查"就是"读代码"
- 没有验证改动的实际效果
- 过度依赖静态分析

### 解决方案
- **动态审查策略**:
  1. 运行测试（包括新增测试）
  2. 启动应用，手动验证改动
  3. 检查日志输出
  4. 测试边界条件
- **对于 bug 修复** - 必须验证 bug 确实被修复
- **对于新功能** - 必须验证功能确实工作

### 预防措施
- 在 SKILL.md 中明确：审查 = 静态分析 + 动态验证
- 要求 AI 输出验证结果

## 失败案例 4: 忽略非功能性问题

### 场景
AI 只关注功能是否正确，忽略了性能、安全、可维护性等问题。

### 表现
- 性能问题（N+1 查询、内存泄漏）
- 安全问题（SQL 注入、XSS）
- 可维护性问题（重复代码、魔法数字）

### 根因
- 只关注"能不能跑"
- 没有考虑非功能性需求
- 缺少安全意识

### 解决方案
- **非功能性审查清单**:
  - **性能**:
    - [ ] 是否有 N+1 查询？
    - [ ] 是否有不必要的循环？
    - [ ] 是否有内存泄漏风险？
  - **安全**:
    - [ ] 用户输入是否验证？
    - [ ] 是否有 SQL 注入风险？
    - [ ] 是否有 XSS 风险？
    - [ ] 敏感数据是否加密？
  - **可维护性**:
    - [ ] 代码是否清晰易懂？
    - [ ] 是否有重复代码？
    - [ ] 是否有魔法数字？
    - [ ] 命名是否清晰？

### 预防措施
- 在 SKILL.md 中添加非功能性审查清单
- 根据改动类型提供不同的清单

## 失败案例 5: 没有检查回归风险

### 场景
AI 只审查了新增功能，没有考虑对现有功能的影响。

### 表现
- 改动破坏了现有功能
- 测试通过但用户报告问题
- 需要紧急回滚

### 根因
- 没有识别改动的影响范围
- 没有运行回归测试
- 没有检查调用方

### 解决方案
- **回归风险评估**:
  1. 识别改动的影响范围（哪些模块会受影响？）
  2. 查找所有调用方（使用 `grep` 或 LSP）
  3. 运行相关的回归测试
  4. 手动验证关键路径
- **高风险改动**:
  - 修改公共 API
  - 修改数据库 schema
  - 修改配置文件
  - 修改核心业务逻辑

### 预防措施
- 在 SKILL.md 中添加回归风险评估
- 对于高风险改动，要求更严格的审查

---

## 失败案例 6: 没有检查文档影响

### 场景
代码改动了 API 或行为，但没有更新文档。

### 表现
- 文档和代码不一致
- 其他开发者困惑
- 用户报告文档错误

### 根因
- 只关注代码，忽略文档
- 认为"文档不重要"
- 没有检查文档影响点

### 解决方案
- **文档影响检查**:
  - [ ] README 是否需要更新？
  - [ ] API 文档是否需要更新？
  - [ ] 注释是否需要更新？
  - [ ] CHANGELOG 是否需要添加条目？
  - [ ] 示例代码是否需要更新？
- **对于 API 改动** - 必须更新 API 文档
- **对于行为改动** - 必须更新 README 或用户文档

### 预防措施
- 在 SKILL.md 中添加文档影响检查
- 要求 AI 列出需要更新的文档

## 反模式总结

### ❌ 不要做
1. **只看 diff** - 忽略上下文和调用方
2. **走过场** - "看起来没问题"
3. **静态审查** - 不运行代码就审查
4. **只关注功能** - 忽略性能、安全、可维护性
5. **忽略回归风险** - 没有考虑对现有功能的影响
6. **忽略文档** - 代码改了但文档没更新

### ✅ 应该做
1. **理解上下文** - 读取完整文件，查找调用方
2. **使用清单** - 逐项检查，输出具体发现
3. **动态验证** - 运行测试，手动验证
4. **全面审查** - 功能 + 性能 + 安全 + 可维护性
5. **评估回归风险** - 识别影响范围，运行回归测试
6. **检查文档影响** - 列出需要更新的文档

## 如何使用本文档

1. **审查前** - 阅读失败案例，准备审查清单
2. **遇到问题时** - 查找类似场景，参考解决方案
3. **审查后** - 如果遇到新的失败案例，添加到本文档
