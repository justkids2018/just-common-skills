# 失败模式记录 (Failure Patterns)

本文档记录 `just-qa` 技能在实际使用中遇到的失败案例和解决方案。

---

## 失败案例 1: 只运行单元测试，忽略集成测试

### 场景
AI 只运行了 `npm test` 或 `pytest`，但没有运行集成测试或 E2E 测试。

### 表现
- 单元测试通过
- 部署到测试环境后发现集成问题
- 需要回滚和修复

### 根因
- 没有识别项目中有多个测试套件
- 只运行了最简单的测试命令
- 没有检查 CI 配置中运行了哪些测试

### 解决方案
- **检查所有测试套件**:
  1. 查看 `package.json` 或 `Makefile` 中的测试命令
  2. 查看 `.github/workflows` 中 CI 运行的测试
  3. 运行所有相关测试
- **分层测试策略**:
  - 单元测试（快速反馈）
  - 集成测试（验证模块协作）
  - E2E 测试（验证完整流程）

### 预防措施
- 在 SKILL.md 中明确：必须运行项目中所有测试套件
- 提供测试发现清单

## 失败案例 2: 测试失败后直接修改代码，没有分析根因

### 场景
测试失败，AI 看了错误信息后立即修改代码，但没有真正理解为什么失败。

### 表现
- 修复了表面问题，但根因还在
- 同样的问题在其他地方再次出现
- 引入了新的 bug

### 根因
- 急于修复，没有耐心分析
- 只看错误信息，没有看堆栈跟踪
- 没有理解测试的意图

### 解决方案
- **失败分析流程**:
  1. 阅读完整的错误信息和堆栈跟踪
  2. 理解测试的意图（测试什么场景？）
  3. 定位根因（为什么这个场景失败？）
  4. 修复根因（不是绕过测试）
  5. 验证修复（测试通过 + 理解为什么通过）
- **使用 `just-investigate`** - 如果根因不明显

### 预防措施
- 在 SKILL.md 中强调：先分析，再修复
- 要求 AI 输出根因分析

## 失败案例 3: 手动验证不充分

### 场景
测试通过后，AI 没有进行手动验证就认为功能正常。

### 表现
- 自动化测试覆盖不全
- 用户体验问题（UI 错位、交互不流畅）
- 边界条件未测试

### 根因
- 过度依赖自动化测试
- 没有启动应用实际操作
- 没有测试边界条件

### 解决方案
- **手动验证清单**:
  - [ ] 启动应用（dev server）
  - [ ] 测试正常路径（happy path）
  - [ ] 测试边界条件（空输入、超长输入、特殊字符）
  - [ ] 测试错误处理（网络错误、权限错误）
  - [ ] 检查 UI/UX（如果是前端改动）
- **对于 UI 改动** - 必须在浏览器中验证
- **对于 API 改动** - 使用 curl 或 Postman 验证

### 预防措施
- 在 SKILL.md 中添加手动验证要求
- 根据改动类型提供不同的验证清单

## 失败案例 4: 构建失败但没有检查构建产物

### 场景
运行 `npm run build` 或 `make build` 成功，但没有检查生成的产物是否正确。

### 表现
- 构建成功但产物不完整
- 部署后发现资源缺失
- 生产环境报错

### 根因
- 只看构建命令的退出码
- 没有检查构建输出
- 没有验证产物内容

### 解决方案
- **构建验证清单**:
  - [ ] 构建命令成功（退出码 0）
  - [ ] 检查构建输出（有没有警告？）
  - [ ] 验证产物存在（dist/ 或 build/ 目录）
  - [ ] 检查产物内容（文件大小合理？）
  - [ ] 运行产物（如果可能）
- **对于前端项目** - 检查 bundle 大小，确保没有意外增大
- **对于后端项目** - 检查二进制文件或 Docker 镜像

### 预防措施
- 在 SKILL.md 中添加构建验证步骤
- 提供产物检查清单

## 失败案例 5: 修复循环 - 改了又改，越改越乱

### 场景
测试失败 → 修复 → 测试还是失败 → 再修复 → 引入新问题 → 继续修复...

### 表现
- 陷入修复循环
- 代码越来越乱
- 不知道哪个改动导致的问题

### 根因
- 没有增量验证
- 一次改动太多
- 没有及时回滚

### 解决方案
- **三次规则** - 如果同一个问题修复了 3 次还没解决：
  1. 停止继续修复
  2. 回滚到最后一个工作状态
  3. 使用 `just-investigate` 深入分析
  4. 重新设计修复方案
- **增量修复** - 一次只改一个地方，立即验证
- **保持 git 干净** - 每次修复后提交，方便回滚

### 预防措施
- 在 SKILL.md 中添加"三次规则"
- 要求 AI 在陷入循环时主动停止

## 失败案例 6: 忽略警告信息

### 场景
测试通过，但有大量警告信息（deprecation warnings, lint warnings）。

### 表现
- 警告积累
- 未来升级困难
- 潜在的运行时问题

### 根因
- 只关注错误，忽略警告
- 认为"警告不重要"
- 没有检查警告内容

### 解决方案
- **警告处理策略**:
  - **Deprecation warnings** - 记录到 TODO，计划修复
  - **Lint warnings** - 立即修复（通常很简单）
  - **Type warnings** - 必须修复（类型安全）
  - **Security warnings** - 高优先级修复
- **零警告原则** - 新代码不应引入新警告

### 预防措施
- 在 SKILL.md 中要求检查警告
- 提供警告分类和处理策略

## 反模式总结

### ❌ 不要做
1. **只运行部分测试** - 漏掉集成测试或 E2E 测试
2. **急于修复** - 没有分析根因就改代码
3. **跳过手动验证** - 过度依赖自动化测试
4. **忽略构建产物** - 只看构建命令是否成功
5. **陷入修复循环** - 改了又改，越改越乱
6. **忽略警告** - 只关注错误

### ✅ 应该做
1. **运行所有测试** - 单元、集成、E2E
2. **先分析，再修复** - 理解根因
3. **手动验证** - 启动应用，实际操作
4. **检查构建产物** - 验证内容和大小
5. **三次规则** - 修复 3 次不成功就停下来分析
6. **处理警告** - 不要让警告积累

## 如何使用本文档

1. **QA 前** - 阅读失败案例，准备验证清单
2. **遇到问题时** - 查找类似场景，参考解决方案
3. **QA 后** - 如果遇到新的失败案例，添加到本文档
