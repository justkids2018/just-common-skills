# 失败模式记录 (Failure Patterns)

本文档记录 `just-careful` 技能在实际使用中遇到的失败案例和解决方案。

---

## 失败案例 1: 没有识别出破坏性操作

### 场景
AI 执行了破坏性命令（如 `rm -rf`、`git reset --hard`），但没有事先警告用户。

### 表现
- 数据丢失
- 无法恢复
- 用户信任度下降

### 根因
- 没有识别命令的破坏性
- 过度自信
- 没有检查影响范围

### 解决方案
- **破坏性操作清单**:
  - **文件删除**:
    - `rm -rf`
    - `git clean -fd`
    - `docker system prune -a`
  - **Git 操作**:
    - `git reset --hard`
    - `git push --force`
    - `git branch -D`
  - **数据库操作**:
    - `DROP TABLE`
    - `TRUNCATE`
    - `DELETE FROM ... WHERE 1=1`
  - **系统操作**:
    - `kill -9`
    - `shutdown`
    - `reboot`
- **执行前必须**:
  1. 识别操作的破坏性
  2. 评估影响范围
  3. 询问用户确认
  4. 提供回滚方案

### 预防措施
- 在 SKILL.md 中添加破坏性操作清单
- 要求 AI 在执行前必须确认

## 失败案例 2: 没有提供回滚方案

### 场景
AI 执行了高风险操作，但没有提前准备回滚方案。

### 表现
- 操作失败后无法恢复
- 需要紧急修复
- 影响生产环境

### 根因
- 只考虑成功路径
- 没有准备失败处理
- 过度自信

### 解决方案
- **回滚方案清单**:
  - **文件操作** - 备份文件：`cp file file.backup`
  - **Git 操作** - 记录当前 commit：`git rev-parse HEAD`
  - **数据库操作** - 备份数据：`mysqldump` 或事务
  - **配置修改** - 备份配置文件
  - **部署操作** - 保留上一个版本
- **执行前必须**:
  1. 准备回滚方案
  2. 告知用户回滚方案
  3. 验证回滚方案可行
  4. 执行操作
  5. 如果失败，立即回滚

### 预防措施
- 在 SKILL.md 中要求：高风险操作必须有回滚方案
- 提供回滚方案模板

## 失败案例 3: 在生产环境执行危险操作

### 场景
AI 在生产环境执行了应该在测试环境执行的操作。

### 表现
- 生产数据被修改
- 服务中断
- 用户受影响

### 根因
- 没有识别当前环境
- 没有区分生产和测试
- 没有环境保护机制

### 解决方案
- **环境识别**:
  1. 检查环境变量：`echo $ENV` 或 `echo $NODE_ENV`
  2. 检查配置文件：`config/production.yml`
  3. 检查主机名：`hostname`
  4. 检查数据库连接：生产 DB vs 测试 DB
- **生产环境规则**:
  - **禁止直接修改数据** - 必须通过 API 或管理工具
  - **禁止删除操作** - 软删除而非硬删除
  - **禁止重启服务** - 除非紧急情况
  - **禁止修改配置** - 必须通过 CI/CD
- **执行前必须**:
  1. 确认当前环境
  2. 如果是生产环境，询问用户确认
  3. 提供更安全的替代方案

### 预防措施
- 在 SKILL.md 中添加环境识别步骤
- 对生产环境操作要求更严格的确认

## 失败案例 4: 没有评估影响范围

### 场景
AI 执行了操作，但没有评估会影响多少数据或多少用户。

### 表现
- 影响范围超出预期
- 大量用户受影响
- 需要紧急回滚

### 根因
- 没有先查询影响范围
- 没有使用 `--dry-run` 或预览
- 过度自信

### 解决方案
- **影响范围评估**:
  - **文件操作** - 先 `ls` 或 `find` 查看会影响哪些文件
  - **数据库操作** - 先 `SELECT COUNT(*)` 查看会影响多少行
  - **Git 操作** - 先 `git diff` 查看会改动哪些文件
  - **部署操作** - 先查看会影响哪些服务
- **使用预览模式**:
  - `git add --dry-run`
  - `rsync --dry-run`
  - `terraform plan`
  - `kubectl apply --dry-run`
- **执行前必须**:
  1. 评估影响范围
  2. 告知用户影响范围
  3. 如果影响范围大，询问确认
  4. 执行操作

### 预防措施
- 在 SKILL.md 中要求：必须先评估影响范围
- 提供影响范围评估清单

## 失败案例 5: 没有检查前置条件

### 场景
AI 执行了操作，但没有检查前置条件是否满足。

### 表现
- 操作失败
- 系统状态不一致
- 需要手动修复

### 根因
- 假设前置条件已满足
- 没有验证系统状态
- 过度自信

### 解决方案
- **前置条件清单**:
  - **文件操作** - 文件是否存在？权限是否足够？
  - **Git 操作** - 工作区是否干净？是否在正确的分支？
  - **数据库操作** - 连接是否正常？权限是否足够？
  - **部署操作** - 依赖服务是否正常？配置是否正确？
- **检查命令**:
  - `test -f file` - 文件是否存在
  - `git status --short` - 工作区是否干净
  - `mysql -e "SELECT 1"` - 数据库是否可连接
  - `curl -f http://service/health` - 服务是否正常
- **执行前必须**:
  1. 列出前置条件
  2. 逐个检查
  3. 如果不满足，先满足前置条件
  4. 执行操作

### 预防措施
- 在 SKILL.md 中要求：必须检查前置条件
- 提供前置条件检查清单

## 失败案例 6: 没有记录操作日志

### 场景
AI 执行了高风险操作，但没有记录操作日志。

### 表现
- 无法追溯操作历史
- 出问题时无法定位
- 审计困难

### 根因
- 认为"执行就行"
- 没有意识到日志的重要性
- 没有记录习惯

### 解决方案
- **操作日志模板**:
  ```markdown
  ## 操作记录

  **时间**: 2026-05-25 15:30:00
  **操作者**: Claude Sonnet 4.6
  **环境**: production

  **操作内容**:
  [描述操作]

  **影响范围**:
  - 影响的文件/数据/服务
  - 影响的用户数量

  **前置条件**:
  - [x] 条件 1
  - [x] 条件 2

  **回滚方案**:
  [如何回滚]

  **执行结果**:
  [成功/失败，详细信息]

  **验证**:
  [如何验证操作成功]
  ```
- **记录到文件** - `operations-<date>.md` 或系统日志

### 预防措施
- 在 SKILL.md 中要求：高风险操作必须记录日志
- 提供操作日志模板

## 反模式总结

### ❌ 不要做
1. **不警告就执行** - 破坏性操作没有确认
2. **没有回滚方案** - 失败后无法恢复
3. **不区分环境** - 在生产环境执行危险操作
4. **不评估影响** - 不知道会影响多少数据
5. **不检查前置条件** - 假设条件已满足
6. **不记录日志** - 无法追溯操作历史

### ✅ 应该做
1. **识别破坏性** - 使用破坏性操作清单
2. **准备回滚** - 备份数据，记录状态
3. **识别环境** - 检查环境变量和配置
4. **评估影响** - 使用预览模式，查询影响范围
5. **检查前置条件** - 逐个验证
6. **记录日志** - 使用操作日志模板

## 安全等级

根据操作的风险程度，分为 3 个安全等级：

### 🟢 低风险（可直接执行）
- 读取文件
- 查询数据库（SELECT）
- 查看日志
- 运行测试

### 🟡 中风险（需要确认）
- 修改文件
- 提交代码
- 重启服务（测试环境）
- 修改配置（测试环境）

### 🔴 高风险（需要确认 + 回滚方案）
- 删除文件/数据
- 强制推送
- 重启服务（生产环境）
- 修改配置（生产环境）
- 数据库 schema 变更

## 如何使用本文档

1. **操作前** - 阅读失败案例，评估风险等级
2. **遇到问题时** - 查找类似场景，参考解决方案
3. **操作后** - 如果遇到新的失败案例，添加到本文档
