# 05 — Git 工作流

## 目标

保证版本历史清晰、变更可追踪、交付可回滚。

## 分支与集成规则

1. 主分支保持可发布状态并受保护。
2. 功能开发必须在独立分支完成。
3. 通过评审和验证后再合并到集成分支。
4. 集成分支通过验证后再进入发布分支。

## Commit Message 规范（强制）

### 标题格式

`<type>(<scope>): <summary>`

- `type` 必填，小写。
- `scope` 可选，表示模块或子系统。
- `summary` 必填，中文或英文都可，但必须明确“改了什么”。

### type 使用规则

| type | 适用场景 |
|------|----------|
| feat | 新增用户可感知能力 |
| fix | 缺陷修复 |
| refactor | 重构（不改变外部行为） |
| perf | 性能优化 |
| docs | 文档变更 |
| test | 测试新增或调整 |
| chore | 构建、脚手架、依赖、流程维护 |
| style | 纯格式调整（不改行为） |

### summary 写法规则

1. 使用动宾结构，直接描述动作和对象。
2. 不写模糊词，如 update、misc、optimize code。
3. 建议不超过 72 个字符。

### body 写法规则（建议）

在需要时补充 2 到 4 行：

1. 为什么改（背景/问题）。
2. 改了什么（关键点）。
3. 影响范围（模块/接口/数据）。
4. 验证方式（测试或手工验证）。

### footer 写法规则（建议）

- 关联需求或任务：`refs: <id-or-path>`
- 破坏性变更：`BREAKING CHANGE: <description>`

## 提交注释与粒度规则（强制）

1. 一个提交只做一件事。
2. 提交信息必须表达变更意图和影响范围。
3. 提交必须可独立回滚。
4. 提交正文中的注释必须可审计：说明原因、影响、验证。
5. 禁止以临时、模糊描述作为正式提交信息。

## 正反例

推荐：

- `feat(auth): 增加设备级登录风控校验`
- `fix(payment): 修复重复回调导致的订单状态覆盖`
- `docs(baseline): 补充 AI 架构解耦检查项`

禁止：

- `update`
- `fix bug`
- `misc changes`
- `wip`

## PR 规则

1. PR 必须包含：改动内容、改动原因、验证方式、风险说明。
2. PR 必须可审阅：变更粒度适中，避免大杂烩。
3. 评审意见必须闭环处理后再合并。

## 风险操作规则

1. 改写历史、强制推送、破坏性清理必须显式确认。
2. 所有高风险操作必须有可恢复方案。
3. 未确认前，AI Agent 不得自动执行高风险操作。

## 检查清单

- [ ] 是否在正确分支开发。
- [ ] 提交是否具备清晰语义。
- [ ] PR 是否包含验证与风险说明。
- [ ] 是否涉及高风险操作且已确认。

## 禁止

- ❌ 将多个无关功能混入同一提交。
- ❌ 未验证就直接合并主线。
- ❌ 无回滚预案执行高风险变更。
