# 框架对比：Waza vs Just-Common-Skills

## 核心结论

**Waza (tw93)**: 极简主义的工程习惯框架，8 个精心挑选的技能，专注个人开发者工作流。

**Just-Common-Skills (你的)**: 企业级编排平台，14+ 技能，为团队协作和生产交付设计。

---

## 1. 核心理念

### Waza
- **"你已经知道的工程习惯，变成 Claude 可以执行的技能"**
- 刻意不完整："八个真正重要的习惯技能"
- 反过度规范："作者写的每条规则都会成为天花板"
- 个人开发者视角：优化个人工作流
- 手动串联：无自动触发，用户控制每个转换

### Just-Common-Skills
- **"跨项目可复用技能的单一真源"**
- 全面覆盖：编排器 + 工作器模式（1+7 核心 + 专用技能）
- 显式治理：AGENTS.md 作为规则真源，baseline 标准
- 团队/企业视角：多项目一致性，生产交付
- 工作流编排：显式阶段门禁，检查点恢复

**核心差异**: Waza 优化个人自主性；Just 优化组织一致性。

---

## 2. 架构设计

### Waza: 三部曲结构
```
Kaku (書く) - 写代码
Waza (技)   - 练习惯
Kami (紙)   - 出文档
```

**8 个核心技能**:
- `/think` - 构建前设计挑战
- `/design` - 前端 UI 迭代
- `/check` - 合并前审查 + 发布
- `/hunt` - 系统化调试
- `/write` - 中英文润色
- `/learn` - 6 阶段研究工作流
- `/read` - URL/PDF 内容抓取
- `/health` - Agent 审计

**技能结构**:
```
skills/<name>/
  ├── SKILL.md (主指令)
  ├── references/ (文档、失败模式)
  └── helpers/ (脚本)
```

### Just-Common-Skills: 编排器 + 工作器

**1 个编排器**:
- `just-dev-pipeline` - 端到端 6 步交付工作流

**7 个核心工作器**:
- `just-plan-eng-review` - 实现前架构评审
- `just-qa` - 实现后验证 + 修复
- `just-review` - 提交前代码审查
- `just-ship` - 提交/PR/发布交接
- `just-document-release` - 发布后文档同步
- `just-investigate` - 根因分析
- `just-careful` - 破坏性操作防护

**7 个专用技能**:
- `just-github-workflows` - CI/CD 模板复制
- `just-deploy-release` - 两步部署工作流
- `just-hotspot-generator` - 热点分析
- `just-card-to-json-workflow` - 卡片生产
- `just-value-red-publish` - 投资研究 + 发布
- `just-feature-doc-generator` - 从代码反向生成文档
- `just-dev-pipeline` - 完整功能交付编排

**技能结构**:
```
skills/just-<name>/
  ├── SKILL.md (必须回答 3 个问题)
  └── guide.md (可选)

common-prompt/baseline/
  ├── 01-design-principles.md
  ├── 02-architecture.md
  ├── 03-coding-standards.md
  ├── 04-testing-standards.md
  ├── 05-git-workflow.md
  └── 06-skill-workflow-standards.md
```

**核心差异**: Waza 是扁平的（8 个平级）；Just 是分层的（编排器委托给工作器）。

---

## 3. 治理模型

### Waza
- **最小规则**: "设定清晰目标和约束，然后退后"
- **记录失败**: 每个坑都来自真实失败
- **反模式**: 跨技能护栏（无幻觉路径、无范围蔓延）
- **项目上下文**: `/check` 读取 README/清单/CI 变得项目感知
- **无正式 baseline**: 约定从使用中涌现

### Just-Common-Skills
- **显式层级**: `AGENTS.md > baseline > ARCHITECTURE.md > 个人偏好`
- **强制 baseline**: 6 个文档定义质量标准（SOLID、测试金字塔、git 安全）
- **技能设计门禁**: 每个 SKILL.md 必须回答 3 个问题（做什么/何时触发/完美输出）
- **工作流门禁**: 多阶段任务需要检查点 + 恢复策略
- **验证要求**: 生产前 3 场景测试（正常/边缘/压力）

**核心差异**: Waza 信任涌现；Just 强制合规。

---

## 4. 工作流设计

### Waza: 手动串联
```
设计:  /think → 批准 → 实现 → /check → 合并
调试:  /hunt → 修复 → /check → 发布
研究:  /read → /learn → /write
```
- 用户控制每个箭头
- 无自动触发
- 技能不调用其他技能

### Just-Common-Skills: 编排阶段
```
just-dev-pipeline (6 步):
  1. 需求澄清
  2. 代码/文档分析
  3. 设计 + 任务拆解
  4. 实现 + 迭代
  5. 自审 + 修复
  6. 提交/PR + 文档收口

just-value-red-publish (4 个门禁):
  数据收集 → 框架分析 → 生成报告 →
  小红书文章 + HTML 卡片 → 发布
  (每个门禁需要用户确认)
```
- 编排器管理阶段转换
- 显式检查点/恢复
- 工作器可独立运行或被委托

**核心差异**: Waza 是用户驱动；Just 是工作流驱动。

---

## 5. 分发与安装

### Waza
**关键特征**：统一的安装入口（`npx skills ...`）+ 少量高频命令。

示例：
- `npx skills add tw93/Waza -a claude-code -g -y`
- `npx skills update -g -y`

**为什么看起来干净**：用户只记 1 类命令，不需要关心仓库内部脚本细节。

### Just-Common-Skills
**对齐后的简化入口**：
- 免 npm：`bash <(curl -fsSL https://raw.githubusercontent.com/qisd/just-common-skills/main/scripts/quick-install.sh)`
- npm 短命令：`npm i -g @qisd/just-common-skills && jcs i`
- 项目注入：`jcs inject /path/to/project --force`

**为什么之前显得长**：文档把内部脚本也暴露给用户。

**现在的策略**：对外只保留两个公共动作（安装、注入），其余脚本归入内部维护。

**核心差异**：Waza 是“统一包管理入口”；Just 是“中心化同步 + 统一短命令入口（jcs）”。

## 6. 优缺点分析

### Waza 优势 ✅
1. **低认知负担**: 8 个技能，易于记忆
2. **快速上手**: 最小规则，边做边学
3. **个人自主**: 用户控制每个决策
4. **可移植**: 跨多个 AI 平台工作
5. **实战验证**: 30 天，300+ 会话，7 个项目
6. **失败驱动**: 每个记录的坑都是真实的
7. **不过度工程**: 刻意不完整

### Waza 劣势 ❌
1. **无团队协调**: 技能不强制跨开发者一致性
2. **无生产安全**: 破坏性操作的护栏有限
3. **手动编排**: 用户必须记住工作流序列
4. **无 baseline 标准**: 代码质量依赖个人判断
5. **有限专业化**: 8 个技能无法覆盖领域特定需求（CI/CD、部署、投资研究）
6. **无恢复策略**: 工作流中断时，用户必须手动重建状态

### Just-Common-Skills 优势 ✅
1. **企业就绪**: 跨团队和项目强制一致性
2. **生产安全**: `just-careful` 防护破坏性操作
3. **自动编排**: `just-dev-pipeline` 管理端到端交付
4. **Baseline 标准**: 6 个文档确保代码质量（SOLID、测试、git 安全）
5. **检查点恢复**: 工作流可在中断后恢复
6. **领域专业化**: CI/CD、部署、投资研究等技能
7. **多项目同步**: 符号链接模式即时传播更新
8. **Copilot 兼容**: 与 GitHub Copilot + Claude Code 协同工作

### Just-Common-Skills 劣势 ❌
1. **高认知负担**: 14+ 技能，复杂层级
2. **陡峭学习曲线**: 必须理解编排器/工作器模式、baseline 文档
3. **过度规范风险**: 刚性规则可能���制 AI 模型改进
4. **重治理**: AGENTS.md + 6 个 baseline 文档 + 技能门禁
5. **中心化依赖**: 所有项目依赖共享中心仓库
6. **迭代较慢**: 生产前需要 3 场景验证
7. **可移植性差**: 与 Claude Code + Copilot 生态系统紧密耦合

## 7. 使用场景适配

### 何时选择 Waza
- **独立开发者** 做个人项目
- **快速原型** 速度 > 一致性
- **学习 AI 辅助开发**（低门槛）
- **跨平台需求**（Claude Code、Codex、Pi、Claude Desktop）
- **最小治理** 偏好（信任涌现而非强制）

### 何时选择 Just-Common-Skills
- **团队/企业** 多开发者和项目
- **生产系统** 需要安全护栏
- **受监管行业** 需要审计跟踪和合规
- **复杂工作流**（CI/CD、部署、多阶段交付）
- **领域专业化**（投资研究、卡片生产等）
- **多项目一致性**（跨仓库共享 baseline）

## 8. 哲学差异

| 维度 | Waza | Just-Common-Skills |
|------|------|-------------------|
| **控制** | 用户驱动 | 工作流驱动 |
| **规则** | 最小（涌现） | 显式（强制） |
| **范围** | 个人习惯 | 团队流程 |
| **完整性** | 刻意不完整 | 全面覆盖 |
| **灵活性** | 高（手动串联） | 中（编排阶段） |
| **安全** | 信任用户判断 | 强制护栏 |
| **学习曲线** | 平缓 | 陡峭 |
| **维护成本** | 低（8 技能） | 高（14+ 技能 + baseline） |

## 9. 混合方案建议

如果想结合两者优势：

### 方案 A: Waza 核心 + Just 扩展
- 使用 Waza 的 8 个核心技能做个人工作流
- 添加 Just 的专用技能（CI/CD、部署、投资）作为可选
- 保持治理最小化（无 baseline 强制）

### 方案 B: Just 核心 + Waza 哲学
- 保留 Just 的编排器/工作器架构
- 将 baseline 文档从 6 个减少到 2 个（设计原则 + git 安全）
- 允许技能"设定目标然后退后"而非刚性规则
- 将 3 场景验证改为可选（推荐，非强制）

### 方案 C: 并行轨道
- **个人模式**: 使用 Waza 的 8 个技能做探索/原型
- **团队模式**: 使用 Just 的编排器做生产交付
- 让开发者根据上下文选择

---

## 10. 总结

**Waza** 是个人开发者的**极简工具包**，重视自主性和速度。它是瑞士军刀：8 个工具，易于携带，足以应对大多数任务。

**Just-Common-Skills** 是团队的**企业平台**，需要一致性和安全性。它是专业工作坊：全面工具，有序工作流，质量保证。

没有"更好"之说——它们优化不同约束：
- Waza 优化**个人速度**
- Just 优化**组织可靠性**

你的选择取决于是独自构建还是团队协作，以及优先考虑速度还是安全。
