# 05 — Git 工作流

## 分支规则

1. 主分支保持可发布状态。
2. 功能开发在独立分支完成。
3. 通过评审和验证后再合并。

## Commit 格式（强制）

`<type>(<scope>): <summary>`

| type | 场景 |
|------|------|
| feat | 新增能力 |
| fix | 缺陷修复 |
| refactor | 重构 |
| docs | 文档 |
| test | 测试 |
| chore | 构建/依赖/流程 |

Summary 规则：动宾结构，描述动作和对象，≤72 字符，禁止模糊词（update、misc）。

## 提交粒度（强制）

1. 一个提交只做一件事。
2. 提交信息表达变更意图和影响范围。
3. 提交必须可独立回滚。

## PR 规则

1. 包含：改动内容、原因、验证方式、风险说明。
2. 变更粒度适中，避免大杂烩。
3. 评审意见闭环处理后再合并。

## 风险操作

1. 改写历史/强制推送/破坏性清理必须显式确认。
2. 高风险操作必须有可恢复方案。
3. AI Agent 未确认前不得执行高风险操作。

## 禁止

- ❌ 多个无关功能混入同一提交。
- ❌ 未验证直接合并主线。
- ❌ 无回滚预案执行高风险变更。
