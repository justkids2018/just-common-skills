# 02 — 架构规范

## 目标

建立统一的架构边界与协作规则，确保系统在 AI 协作开发场景下具备可扩展性、可替换性、可测试性和可审计性。

## 分层原则（强制）

1. 领域层：承载业务规则与核心模型。
2. 应用层：承载业务流程编排。
3. 适配层：承载对外协议与外部系统接入。
4. 基础设施层：承载框架、存储、网络与运行时能力。

依赖方向必须单向由外向内，禁止反向依赖和循环依赖。

## 模块边界（强制）

1. 模块对外只暴露契约，不暴露内部实现。
2. 跨模块协作通过接口、事件或 API 契约。
3. 一个模块内保持高内聚，避免跨模块共享业务实现。
4. 允许共享底层能力，不允许共享上层业务逻辑。

## AI 架构补充（强制）

1. Prompt、工具编排、模型配置必须模块内聚管理。
2. 模型后端必须可替换，业务逻辑不得绑定单一供应商。
3. 上下文拼装与工具调用链必须可追踪、可审计。
4. 模型失败、超时、限流必须有显式降级策略。
5. AI 相关安全策略必须前置（数据分级、脱敏、权限控制）。

## 架构偏好（推荐）

1. DDD 优先：先稳定领域实体、值对象和领域服务，再接入外部能力。
2. Clean Architecture 优先：AI 能力作为可替换适配器，不进入核心业务规则。
3. 简洁解耦优先：优先小接口、薄编排、少层级，避免过度抽象。
4. 可读优先：命名清晰、职责单一、结构直观，降低人和 AI 的理解成本。
5. 渐进演进优先：先满足当前确定性需求，再按真实复杂度演进。

## 复杂度控制（强制）

1. 架构设计优先满足当前已确认需求，禁止为未确认场景预建复杂结构。
2. 抽象必须有明确复用或替换收益；无收益抽象不得引入。
3. 跨层调用必须通过定义好的契约，不得通过临时捷径穿透边界。
4. 新增中间层、编排层或共享层时，必须给出必要性说明与替代方案比较。

## 核心域保护与业务独立（强制）

1. 核心域默认受保护，业务功能开发不得直接改动核心域实现。
2. 如需改动核心域，必须先完成需求说明、影响评估和评审审批。
3. 业务功能必须在本业务目录内闭环实现，禁止跨目录散落实现。
4. 跨业务协作必须通过契约（接口、事件、API），禁止跨目录改写内部实现。
5. 共享能力仅承载通用能力，不承载具体业务规则。

## 目录组织参考（非强制）

核心原则：按业务边界组织目录，目录形态可演进，边界不可破坏。

参考示例：

```text
modules/
	customer/
		...
	order/
		...

shared/
	...
```

说明：

1. 同一业务功能下可以按需要继续加子目录（如 api、domain、ai、tests）。
2. 业务目录之间不直接依赖内部实现，协作通过显式契约完成。
3. `shared/` 只放通用能力，不放具体业务规则。
4. 目录命名不做硬编码，重点是边界清晰与职责单一。

## 架构决策要求

1. 重大架构决策必须记录：目标、备选方案、取舍依据、风险、回滚方案。
2. 新增依赖必须说明替代方案比较和引入理由。
3. 架构变更必须评估边界影响与兼容性影响。

## 核心域变更流程（强制）

1. 提交变更前必须给出需求说明、影响评估和风险说明。
2. 变更方案必须经过评审批准后实施。
3. 实施后必须补充变更记录，明确影响范围与回滚路径。

## 检查清单

- [ ] 是否存在反向依赖。
- [ ] 是否存在循环依赖或跨模块直接调用内部实现。
- [ ] 共享层是否承载了业务规则。
- [ ] 新功能是否在不破坏边界的前提下可扩展。
- [ ] 业务功能是否仅在本业务目录内实现（无跨目录改写）。
- [ ] 核心域变更是否具备需求说明、影响评估、评审记录和回滚路径。
- [ ] 业务逻辑是否对单一模型供应商产生硬绑定。
- [ ] AI 调用链是否具备可追踪、可审计能力。
- [ ] 关键失败路径是否具备降级与告警策略。
- [ ] 是否存在未证明收益的抽象层或中间层。
- [ ] 代码组织是否满足可读优先（命名清晰、职责单一、结构直观）。

## 禁止

- ❌ 循环依赖。
- ❌ 在边界层编写业务规则。
- ❌ 在核心层绑定外部框架或供应商能力。
- ❌ 通过临时捷径破坏既有分层和模块边界。
- ❌ 无审计能力的黑盒 AI 调用。
- ❌ 未经评审直接改动核心域。
- ❌ 业务功能跨目录改写其他业务内部实现。
- ❌ 为未确认需求提前引入复杂中间层或过度抽象。
