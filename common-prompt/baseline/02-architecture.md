# 02 — 架构规范

## 目标

建立统一架构边界，保证系统可扩展、可替换、可测试，并适配 AI 场景下的模块化解耦要求。

## 分层原则（强制）

1. 领域层：承载业务规则与核心模型。
2. 应用层：承载业务流程编排。
3. 适配层：承载对外协议与外部系统接入。
4. 基础设施层：承载框架、存储、网络与运行时能力。

依赖方向必须单向由外向内，不允许反向依赖。

## 模块边界（强制）

1. 模块对外只暴露契约，不暴露内部实现。
2. 跨模块协作通过接口、事件或 API 契约。
3. 一个模块内保持高内聚，避免跨模块共享业务实现。
4. 允许共享底层能力，不允许共享上层业务逻辑。

## AI 架构补充（强制）

1. Prompt、工具编排、模型配置必须模块内聚管理。
2. 模型后端必须可替换，业务逻辑不得绑定单一供应商。
3. 上下文拼装与工具调用链必须可追踪、可审计。
4. 模型失败、超时、限流必须有显式降级策略。
5. AI 相关安全策略必须前置（数据分级、脱敏、权限控制）。

## 简单用例（目录区隔）

核心思路：每个业务功能在自己的目录内持续演进，目录可以增减，但边界不要破。

参考写法（非强制模板）：

```text
modules/
	customer/
		...
	order/
		...

shared/
	...
```

说明：

1. 同一业务功能下可以按需要继续加子目录（如 api、domain、ai、tests）。
2. 业务目录之间不直接依赖内部实现，协作通过契约完成。
3. `shared/` 只放通用能力，不放具体业务规则。
4. 目录命名不做硬编码，重点是“按业务边界隔离”。

## 架构决策要求

1. 每个重大架构决策必须写清：目标、取舍、风险、回滚方案。
2. 新增依赖必须说明替代方案比较和引入理由。
3. 架构变更必须评估对现有模块边界的影响。

## 检查清单

- [ ] 是否存在反向依赖。
- [ ] 是否存在跨模块直接调用实现。
- [ ] 是否存在共享层承载业务逻辑。
- [ ] 是否能在不破坏边界的前提下扩展新功能。
- [ ] 是否存在业务逻辑对单一模型供应商的硬绑定。
- [ ] AI 调用链是否具备审计与追踪能力。
- [ ] 关键失败路径是否具备降级与告警策略。

## 禁止

- ❌ 循环依赖。
- ❌ 在边界层写业务规则。
- ❌ 在核心层绑定外部框架或供应商能力。
- ❌ 通过临时捷径破坏既有分层。
- ❌ 无审计能力的黑盒 AI 调用。
