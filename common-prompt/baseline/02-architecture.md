# 02 — 架构规范

## 分层（强制）

1. 领域层 → 应用层 → 适配层 → 基础设施层。
2. 依赖单向由外向内，禁止反向和循环依赖。

## 模块边界（强制）

1. 对外只暴露契约，不暴露内部实现。
2. 跨模块协作通过接口、事件或 API 契约。
3. 共享层只放通用能力，不放业务规则。

## AI 架构（强制）

1. Prompt/工具编排/模型配置模块内聚管理。
2. 模型后端可替换，不绑定单一供应商。
3. AI 调用链可追踪、可审计。
4. 模型失败/超时/限流有显式降级策略。

## 架构偏好

1. DDD 优先：先稳定领域实体再接入外部能力。
2. Clean Architecture：AI 作为可替换适配器，不进核心规则。
3. 简洁解耦：小接口、薄编排、少层级。
4. 可读优先：命名清晰、职责单一、结构直观。
5. 渐进演进：先满足当前需求，按真实复杂度演进。

## 核心域保护（强制）

1. 核心域默认受保护，改动需需求说明 + 影响评估 + 评审。
2. 业务功能在本业务目录内闭环，禁止跨目录改写。
3. 跨业务协作通过契约，不直接改内部实现。

## 复杂度控制

1. 无确认收益的抽象不得引入。
2. 新增中间层必须说明必要性与替代方案。
3. 跨层调用必须通过契约，不走临时捷径。

## 禁止

- ❌ 循环依赖。
- ❌ 在边界层写业务规则。
- ❌ 核心层绑定外部框架。
- ❌ 无审计能力的黑盒 AI 调用。
- ❌ 未经评审改动核心域。
- ❌ 为未确认需求引入过度抽象。
