# 03 — 编码规范

## 目标

提升可读性、可维护性和协作一致性，降低缺陷率。

## 命名与结构

1. 命名必须语义化且一致，禁止模糊缩写。
2. 模块和文件名应与职责一致，避免一文件多职责。
3. 函数应短小、单一职责，避免过深嵌套。

## 复杂度控制

1. 超长文件、超长函数、过多参数必须拆分。
2. 复杂分支逻辑必须重构为可读的结构化流程。
3. 重复逻辑达到稳定重复后再抽象。

## 注释与文档

1. 公共接口必须有用途说明与边界说明。
2. 注释解释设计意图和取舍，不复述代码语法。
3. TODO 必须有责任人和日期。

## 错误与安全

1. 错误必须显式处理，禁止静默吞错。
2. 边界层做输入校验，核心层保证业务约束。
3. 敏感信息不得进入日志与输出。
4. 任何外部输入默认不可信，先校验再使用。

## 检查清单

- [ ] 命名与职责是否一致。
- [ ] 是否存在可拆分的高复杂度函数。
- [ ] 公共接口是否有必要注释。
- [ ] 错误处理是否完整可追踪。
- [ ] 是否存在敏感数据泄露风险。

## 禁止

- ❌ 注释掉废弃代码而不删除。
- ❌ 调试残留进入主分支。
- ❌ 无意义命名和无主 TODO。
- ❌ 将临时修补当成长期方案。
